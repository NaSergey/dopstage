# Принципы организации кода по файлам (SOLID / DRY / KISS)

Как в проекте устроены файлы: **что за код лежит в каком файле и почему**. Это «правила игры» для переноса в новый проект — не про папки (это в [ARCHITECTURE.md](ARCHITECTURE.md)), а про **ответственность файла**.

---

## Главное правило: один файл = одна ответственность (SRP)

Каждый файл отвечает за **одну вещь** и имеет понятное имя-обещание. Если в имени файла можно поставить «и» (`card-and-fetch.tsx`) — это два файла.

| Тип файла | За что отвечает | Чего там НЕ должно быть |
|---|---|---|
| `*.tsx` (компонент) | Разметка + композиция | Фетч данных, сложные вычисления, бизнес-правила |
| `*-server.tsx` | Загрузка данных на сервере (RSC) | Интерактив, `useState`, обработчики |
| `*-client.tsx` | Интерактив + подписка на данные (react-query) | Тяжёлые чистые вычисления (выносить в `lib/`) |
| `*-skeleton.tsx` | Состояние загрузки | Любая логика |
| `use-*.ts` | Один хук — одна задача | Разметка (JSX) |
| `lib/*.ts` | Чистая функция (вход → выход) | React, состояние, side-effects |
| `model/*.ts` | Типы, конфиги, контексты, константы | Разметка |
| `*-store.ts` | Состояние + экшены (zustand) | Запросы, разметка |
| `*-columns.tsx` | Описание колонок таблицы | Сам компонент таблицы |
| `*-icon.tsx` | Одна иконка (SVG) | Что угодно кроме иконки |

## 1. SRP — разделение по оси «данные / интерактив / отображение»

Один UI-блок физически разбит на 3 файла, потому что у них **3 разные причины меняться**:

```
programs-panel/
├── programs-panel.tsx          # SERVER: фетч + revalidate, отдаёт initialData
├── ui/programs-panel-client.tsx# CLIENT: react-query, состояния error/loading
├── ui/programs-panel-skeleton.tsx # отображение загрузки
└── ui/card.tsx                 # чистое отображение одной карточки
```

- **server-файл** меняется, когда меняется источник/кэширование данных.
- **client-файл** — когда меняется логика обновления/ошибок.
- **card / skeleton** — когда меняется внешний вид.

Карточка ([card.tsx](src/app/(main)/ui/programs-panel/ui/card.tsx)) принимает готовые `data` через props и **только рисует** — не знает, откуда данные. Это и есть SRP + слабая связанность.

## 2. SRP для функций: одна функция — один файл (в `lib/` и `format/`)

Чистые хелперы лежат по одному в файле, имя файла = имя функции:

```
shared/lib/format/
├── formatNumber.ts   → formatNumber()
├── formatK.ts        → formatK()      (использует formatNumber — DRY)
├── formatAddress.ts  → formatAddress()
├── getTimeAgo.ts     → getTimeAgo()
└── toPercent.ts      → toPercent()
```

`formatK` строится на `formatNumber` ([formatK.ts](src/shared/lib/format/formatK.ts)) — переиспользование, а не копипаста. Так функцию легко найти, протестировать и заменить.

**Правило:** если функция не зависит от React и не делает side-effects — её место в `lib/`, отдельным файлом, и её можно дёрнуть откуда угодно.

## 3. KISS — «глупые» компоненты, логика снаружи

Компонент должен читаться сверху вниз без расшифровки. Сложность выносится:

- **Тяжёлые вычисления → `lib/`.** Пример: настройка графика на 70 строк живёт в [create-receptor-chart.ts](src/app/(main)/ui/receptors/ui/chart/lib/create-receptor-chart.ts), а компонент `chart.tsx` просто вызывает `createReceptorChart(container)`. Компонент не раздут конфигом.
- **Логика запроса → `hooks/use-*.ts`.** Пример: [use-top-tweets.ts](src/app/subnet/[id]/components/top-tweets/hooks/use-top-tweets.ts) прячет `useInfiniteQuery`, склейку страниц и состояние ошибки. Компонент получает готовые `{ data, isLoading, fetchNextPage, ... }`.
- **Магические числа → именованные константы.** `CHART_STYLES` в начале файла, `PAGE_SIZE = 20`, `LONG_REFETCH_INTERVAL`. Никаких `300000` прямо в JSX.

Маленькую локальную функцию-форматтер допустимо держать рядом с компонентом, если она нужна только ему:
```ts
const formatPercentage = (n: number) => (n <= 5 ? n.toFixed(2) : n.toFixed(1));
```
(см. [card.tsx](src/app/(main)/ui/programs-panel/ui/card.tsx)) — KISS важнее догмы «всё в lib».

## 4. DRY — единые точки правды

Дублирование убирается через **один модуль-источник**:

- **Типы данных — из API-схемы, не вручную.** `ApiComponents["TopProgramResponse"]` вместо самописных интерфейсов. Поменялся бэкенд → перегенерили `generated.ts` → типы обновились везде.
- **Обработка ошибок react-query — один слой.** [shared/lib/error](src/shared/lib/error/index.ts) даёт `refetchInterval`, `refetchOnReconnect`, `useErrorState`. Компоненты импортируют одной строкой и не повторяют логику «на 404 не рефетчить».
- **Тайминги — константы + env.** [refetchInterval.ts](src/shared/lib/constants/refetchInterval.ts) + `NEXT_PUBLIC_*_INTERVAL`. Одно место правит частоту обновления.
- **Маппинг URL↔API — один файл.** [leaderboard-params.ts](src/app/leaderboard/lib/leaderboard-params.ts): словари `FRONTEND_PARAMS`/`API_PARAMS` + `toApiParams`, чтобы не хардкодить имена параметров.
- **`cn()` для классов** ([css.ts](src/shared/lib/utils/css.ts)) — один способ склеивать Tailwind-классы во всём проекте.
- **Барель-файлы** (`icons/index.ts`, `error/index.ts`) — единая точка импорта, чтобы потребителю не знать внутреннюю раскладку.

## 5. Open/Closed — расширение через props и варианты, а не правкой

Компоненты расширяются, не переписываясь:

- **Варианты вместо новых компонентов:** `<ProgressBar variant="dense" />`, `<AppImage fallbackVariant="octagon" />`, кнопки на `class-variance-authority` (cva). Новый вид = новый вариант, старый код не трогаем.
- **Поведение через props-флаги:** `<ProgramsLeaderboardTableServer showSeeAllButton pageSize={20} />`. Дефолты заданы, вызов переопределяет точечно.
- **`data-table` обобщённый**, конкретика — снаружи: `*-columns.tsx` описывает колонки, таблица их рендерит. Новая таблица = новый файл колонок, движок таблицы не меняется.

---

## 6. ISP / слабая связанность — узкие props, данные сверху вниз

- Компонент принимает **ровно то, что рисует**: `type Props = { data: ApiComponents["TopProgramResponse"] }`. Не получает весь стор или весь ответ API «на всякий случай».
- **Данные текут вниз через props, состояние — вверх/в стор.** `*-server` → `initialData` → `*-client` → дочерние «глупые» компоненты.
- Хук возвращает **готовый интерфейс потребления** (`{ data, hasError, showReload, refetch }`), скрывая, что под капотом react-query. Потребитель не зависит от деталей реализации.

---

## 7. DIP — зависимость от абстракций (клиенты, не fetch)

- Компоненты не зовут `fetch` напрямую — только `rqClient` (клиент) или `serverFetchClient` (сервер). Сменить транспорт/базовый URL/заголовки можно в одном месте (`instance.ts` / `server.ts`), компоненты не трогаются.
- Базовый URL и тайминги приходят из env (`NEXT_PUBLIC_*`), а не зашиты в коде.

## 8. Состояние — по месту использования

KISS-правило выбора хранилища:

| Что | Где |
|---|---|
| Серверные данные (ответы API) | React Query (кэш) |
| Глобальное клиентское состояние | zustand-store в `features/` (напр. `auth-store.ts`) |
| Состояние одной страницы | локальный `store/*-store.ts` рядом с маршрутом |
| Локальный UI внутри слайса | React Context в `model/` |
| Фильтры/сортировки списков | URL-параметры |

Стор держит **инварианты в экшенах**, а не размазывает их по компонентам (напр. в `auth-store.ts` logout юзера сбрасывает и кошелёк).

---

## 9. Размещение файла — рядом с использованием

Принцип близости (colocation): файл лежит максимально близко к тому, кто его использует, и поднимается в общий слой только когда нужен в ≥2 местах.

```
нужно одной странице   → src/app/<route>/components|hooks|lib|store/
нужно одной фиче        → src/features/<feature>/...
нужно везде             → src/shared/...
```

Не выносим в `shared` то, что используется в одном месте (это усложняет, нарушает KISS). И наоборот — заметили копипасту в 2+ местах → поднимаем в общий слой (DRY).

## Чек-лист «куда положить новый код»

1. Это чистая функция без React? → `lib/<name>.ts`, отдельный файл, имя = функция.
2. Это логика данных/состояния для компонента? → `hooks/use-*.ts`.
3. Это загрузка данных на сервере? → `*-server.tsx` (только фетч + `initialData`).
4. Это интерактив/живые данные? → `*-client.tsx` с `"use client"`.
5. Это только разметка? → обычный `*.tsx`, данные через props, никакого фетча.
6. Это тип/конфиг/контекст/константа? → `model/*.ts`.
7. Это глобальное/страничное состояние? → `*-store.ts` (zustand+immer).
8. Это колонки таблицы? → `*-columns.tsx`, отдельно от таблицы.
9. Магическое число? → именованная константа (или env, если конфигурируемо).
10. Уже встречал такой код? → не копируй, вынеси в общий модуль и импортируй.

> Короткая мантра проекта: **данные — в server/hook, вычисления — в lib, отображение — в tsx, типы — из схемы, повтор — в shared.**
